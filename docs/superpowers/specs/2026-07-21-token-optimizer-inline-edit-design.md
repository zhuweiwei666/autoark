# Token 优化师行内编辑设计

## 目标

允许用户在“Token 与像素”页面的 Token 列表中手动标记或修改对应优化师，并将结果持久保存。编辑操作不得暴露或修改 Facebook Token。

## 方案选择

采用表格行内编辑：操作列增加“编辑”，点击后当前行的优化师单元格切换为输入框，并显示“保存 / 取消”。相比弹窗，行内编辑适合连续处理多条 Token；相比系统用户下拉，自由输入也能覆盖外部投手。

## 交互

- 未编辑时显示优化师名称；空值显示 `-`。
- 点击“编辑”后，仅当前行进入编辑状态，输入框预填原值并自动聚焦。
- 按 Enter 或点击“保存”提交；按 Escape 或点击“取消”放弃本次修改。
- 输入在保存前去除首尾空格，最多 80 个字符；空字符串表示清除标记。
- 保存期间禁用当前行的保存、取消、检查和删除操作，防止重复提交或并发冲突。
- 保存成功后退出编辑状态、刷新 Token 列表并显示“优化师已更新”。
- 保存失败时保留编辑状态和用户输入，并显示接口错误，便于直接重试。

## 数据流与权限

- 前端复用现有 `updateToken(id, { optimizer })` 调用。
- 后端现有 `PUT /api/fb-token/:id`、字段清洗、80 字符上限和租户作用域保持不变。
- 空值显式发送为空字符串，以触发现有清除逻辑。
- 本次只更新 Token 的归属标记，不立即批量覆盖账户目录。账户可能被多个 Token 访问，立即级联会产生归属歧义；后续正常账户同步仍会按既有规则带入 Token 优化师。

## 测试与验收

- 回归测试覆盖：当前页面导入更新 API、存在行内编辑入口、空值可清除、Enter 保存、Escape 取消、保存失败保留编辑状态、保存期间禁止重复操作。
- 前端 TypeScript 检查和生产构建必须通过。
- 后端完整测试保持通过，确认未破坏既有 Token 更新与作用域能力。
- 生产环境验证一条 Token：进入编辑、保存名称、刷新后仍保留；随后恢复原值，避免留下测试数据。

## 非目标

- 不新增优化师字典或系统用户绑定。
- 不修改 Facebook Token、OAuth、像素同步和账户归属模型。
- 不对已有账户做无条件批量级联更新。
